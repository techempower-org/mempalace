# Three MemPalace Consumers: Convergent Substrate, Divergent Intelligence Layers

*Field notes on three independent production-shaped systems building on MemPalace, what they agree on, where they diverge, and what the triangulation suggests about agent memory architecture in mid-2026.*

## The three systems

In the span of a single conversation about agent memory architecture, three production-shaped systems surfaced — all built on MemPalace as the underlying memory substrate, all developed independently by different practitioners in different domains, all addressing variants of what the main writeup called the consumption problem (high retrieval recall but uneven end-to-end task performance). The convergences and divergences across these three systems are worth mapping explicitly because they suggest something about where the architectural choices actually live.

**Familiar** ([github.com/jphein/familiar.realm.watch](https://github.com/jphein/familiar.realm.watch)) is a deterministic retrieval pipeline running local Gemma against jphein's MemPalace fork. Rerank, temporal decay, extractive compression, grounding directives applied to vector + BM25 hybrid retrieval, all running unconditionally on every turn. Domain: general agent use, coding/dev work. Measured outcome: 78.33% recall on the jp-realm-v0.1 corpus versus 46.67% for RLM-orchestrated retrieval at the same task.

**CampaignGenerator** ([github.com/kostadis/CampaignGenerator](https://github.com/kostadis/CampaignGenerator)) with kostadis/mempalace fork — a tabletop RPG session-prep system over a 2 TB local PDF library plus the 5etools JSON corpus. Rank-bucketed AAAK projections at wing and room levels enable hierarchical pruning before drawer-level vector search runs. Domain: GM tooling with hard verbatim-attribution requirements. Measured outcome: 19.82× cost reduction at 0% recall@10 loss versus flat search on a 281-entry benchmark fixture.

**Kent** ([github.com/kenchambers/kent](https://github.com/kenchambers/kent)) is a typed async agent runtime with default-on persistent memory plus a training subsystem. APO (Automatic Prompt Optimization) via Microsoft Agent Lightning optimizes five prompt resources against critic-scored rollouts. Domain: general-purpose terminal agent with Discord gateway and 3D palace viewer. Measured outcome: APO Round 01 completes with v0=0.866 baseline; drawer-aware query embedding similarity averages 0.323 versus 0.027 for unrelated queries (3/3 pairwise wins).

These are different projects in different domains by different people. They share a substrate and very little else.

## What all three agree on

Four agreements emerge across all three systems. None are surprising in isolation; the fact that all three made the same choice without coordination is informative.

**Verbatim storage is the right base layer.** None of the three systems summarizes content on the way in. Familiar relies on MemPalace's raw drawers (ChromaDB → Postgres+pgvector+AGE in jphein's migration). CampaignGenerator stores 5etools entities and adventure prose verbatim, indexing facets in metadata but never paraphrasing the content. Kent's `sweeper.sweep()` path writes turn transcripts as raw JSONL ingested verbatim. All three explicitly reject upstream summarization — when Kostadis spells out why, he notes that LLM-driven intermediate compression introduces "drift" (same content scores differently across runs) and "recall loss at the prune step" (a summary that drops a token makes the underlying drawer unreachable). The hard recall requirement is load-bearing across all three.

**The LLM is not in the index path.** Familiar's pipeline is fully deterministic — rerank, temporal decay, extractive compression are all rule-based or trained-model stages, not LLM calls. Kostadis explicitly forbids LLMs in the index or pruning path; the wing index for query `q` is a pure function of the leaf closets under that wing. Kent's `memory_recall` tool routes to `Layer3.search` which does straight vector search; the LLM only consumes results. None of the three uses RLM-style depth-N recursion for retrieval. The "Think, But Don't Overthink" finding from the RLM reproduction work — that depth>1 recursion collapses format and explodes latency — appears to have landed across all three independently, before the reproduction paper would have been the obvious citation.

**Wings (or equivalents) provide scope, not classification.** All three use named contexts to scope memory but treat scope as best-effort routing rather than required classification. Familiar inherits MemPalace's wings but uses them as optional metadata filters. CampaignGenerator routes 5etools entities into typed wings (`wing_bestiary`, `wing_spells`) but allows entity facets to drive retrieval more than wing membership. Kent maps Discord channels to wings automatically per-channel — wing assignment is a side effect of the conversational context, not a decision the agent makes. None of the three treats wing taxonomy as the primary organizational principle the way the original MemPalace palace metaphor implies. The MemPalace fork README's "Stop investing in hierarchy; invest in retrieval" lands here.

**The consumption problem is real and not solved by retrieval quality.** All three systems independently add layers above the retrieval substrate specifically to address what happens after a drawer is found. Familiar adds rerank and grounding directives. CampaignGenerator adds the retrieve/render isolation CI invariant and cost-tagged candidates. Kent adds APO-trained query rewriting and the recall self-improvement games. Three different mechanisms answering the same question: how do we get from "98% recall@5" to actually-useful task performance?

## Where they diverge

Three sharp divergences. Each represents a different bet about where the intelligence above the verbatim substrate should live.

**Familiar: intelligence in the algorithm.** The retrieval pipeline runs the same fixed sequence on every query — semantic + BM25 → cross-encoder rerank → temporal decay → extractive compression → grounding directives. No LLM decides anything during retrieval; no human reviews candidates between retrieval and rendering. The bet: a sufficiently good deterministic pipeline running unconditionally beats both LLM-orchestrated and human-in-the-loop variants because it sidesteps the invocation problem entirely. The pipeline is always on, so the agent never has to decide whether to call it. Cost: no adaptation to task structure; the same pipeline runs whether the query is single-hop factual recall or multi-hop synthesis. Whether that limitation matters depends on how much the workload actually varies.

**CampaignGenerator: intelligence in the human-in-the-loop.** Kostadis's retrieve/render isolation invariant is enforced as a static-analysis CI check that fails the build if any Python function contains both a retrieval call and a render call. The discipline is mechanical: the human GM reviews retrieved candidates, marks them approved or removes them, and only then can a separate process render text. The bet: scope decisions ("which sources are authoritative for *this* question?") are precision decisions the human should make explicitly, because the cost asymmetry between a wrong drawer fetched (cheap re-query) and a wrong drawer rendered into prose read aloud at the table (session-derailing error) is too steep to delegate. Cost: friction. Every render requires a human checkpoint. Won't scale to use cases where the human can't or won't review per turn.

**Kent: intelligence in the trained policy.** APO optimizes five prompt resources — `query_rewrite_policy`, `scope_policy`, `closet_summary_policy`, `actor_system_prompt`, `retrieval_policy` — against rollouts scored by a critic LLM. The "recall games" (Game A: recall@k; Game B: scope selection; Game C: closet fidelity; Game D: tunnel utility) are self-improvement loops where the agent practices recalling its own palace under different conditions. The bet: the consumption problem is a policy-learning problem, addressable by training the prompts that govern memory invocation behavior rather than by hard-coding pipelines or requiring human review. Cost: dependency on critic quality (Kent ships cross-critic consensus and collusion probes specifically to address this), and the algorithm is still early — APO Round 01 in the live test produced v1=0.778 from v0=0.866, a regression in one round. The optimization machinery works; whether it discovers actually-better prompts over multiple rounds remains the open question Kent's "suggested next tests" specifically calls out.

The three bets are not mutually exclusive. Familiar's pipeline could be the deterministic ground layer; Kostadis's human checkpoint could sit at the render boundary; Kent's APO could optimize prompts for both. But no system in the wild combines all three, and the choices about which layer earns its complexity remain unresolved.

## Patterns each surfaces that the others don't

Beyond the high-level divergence, each system surfaces specific engineering patterns worth pulling out individually.

**From Familiar:** the "always-on pipeline" pattern as an answer to invocation failures. The 32-point recall gap between Familiar (78.33%) and RLM with the same underlying retrieval (46.67%) comes entirely from invocation — RLM agents zero-call the tool on 22-25 of 30 questions; Familiar's pipeline runs on all 30 because the agent doesn't choose. When the SME data lands publicly, this will likely be the most-cited single result from the conversation.

**From CampaignGenerator:** the cost-tagged multi-source candidate pattern. The retriever consults three awareness sources (palace, 5etools catalog, rpg-library HTTP) and returns merged candidates with `kind / cost` discriminators — cheap candidates carry a one-line ingest command, expensive candidates carry a conversion + ingest pair. The retriever doesn't decide whether to materialize uningested content; it surfaces both as candidates with explicit costs. For domains with large unconverted corpora (legal, scientific PDFs, code archives), this is more practical than bulk ingest. None of the OSS memory systems in the broader landscape exposes this surface.

**From Kent:** three patterns worth naming separately. *Wake-up at compaction* — Kent embeds L0+L1 essential-moment summaries inline in the message that `maybe_compact` produces, so priming refreshes after every compaction instead of being lost when the head is summarized away. *Heartbeat agent* — periodic agent turns triggered by a free-form Markdown file (`~/.kent/HEARTBEAT.md`) on configurable cadence; each tick is a full agent turn the agent can use to write diary entries, run searches, or post to Discord. Memory maintenance happens between conversations, not just during them. *Channel-as-wing automatic routing* — Discord channels map to wings deterministically (`discord_<guild>_<channel>`), so wing assignment is a side effect of conversational context rather than an agent decision, sidestepping a class of routing-decision failure modes.

Each of these patterns is small in isolation. They're worth naming because they're the kind of operational engineering that doesn't show up in benchmark numbers but matters enormously in a system someone actually uses for months.

## Open questions all three share

Three problems neither Familiar nor CampaignGenerator nor Kent has solved.

**Concurrent writes under load.** All three flag concurrent-write safety as unverified or limited. Kent's Known Limitations explicitly notes that sibling subagents writing to the same JSONL transcript and ChromaDB upsert path concurrently may race. Kostadis's piece notes the graph cache uses 60s TTL with write invalidation but doesn't claim it's robust under high concurrency. jphein's MemPalace fork README documents file-level locking and HNSW mtime detection as workarounds for ChromaDB's lack of transactional storage (the Postgres+pgvector+AGE migration is meant to address this). Three independent systems hitting the same underlying issue suggests the MemPalace concurrency model needs work — file-level locking and mtime-based cache invalidation are not the right primitives for systems that need parallel-safe writes.

**When does ad-hoc decomposition pay off?** Kostadis names this as the most interesting unexplored direction: an RLM-style sub-call for queries where the wing taxonomy doesn't cleanly fit. Kent's adaptive `max_depth` would land in similar territory. Familiar's fixed pipeline has no analog. None of the three currently has a heuristic for "should I run my deterministic pipeline, or should I escalate to an LLM-orchestrated decomposition?" The break-even insight from the RLM paper (RLM helps complex tasks, hurts simple retrieval) is acknowledged but unmeasured in any of these production-shaped systems.

**Cross-system benchmarking.** All three measure themselves against their own corpora and their own baselines. Familiar's 78.33% vs RLM's 46.67% on jp-realm-v0.1. CampaignGenerator's 19.82× cost reduction at 0% recall loss on a 281-entry RPG benchmark. Kent's recall@k responding to query quality on its training set. None of these numbers is directly comparable across systems because the workloads differ. A shared benchmark spanning verbatim-first systems would let the field measure these architectural patterns against each other rather than each against its own internal baseline. SME is the closest thing to that benchmark in flight, and it covers Familiar and RLM but not the others.

## What the triangulation suggests

Three observations that follow from the convergence rather than from any single system.

**Verbatim storage with deterministic intermediate compression is a real architectural pattern, not an idiosyncrasy.** The pattern has now appeared independently in three production-shaped systems across general agent use, RPG session prep, and Discord-bot memory. The consistency across domains is notable. It contrasts with the compile-upstream architectures (Pinecone Nexus, Cognee, Letta) that bundle ingestion-time LLM processing as the primary design choice. The bet "preserve raw content, add intelligence above" appears to win in domains where exact attribution and recall floor matter. Whether it scales to domains where summarization is the actual user need is a separate question none of these three answers.

**The intelligence layer is unsettled and probably should be.** Familiar's algorithm, Kostadis's human-in-the-loop, and Kent's trained policy are three different answers to the same question. None is dominant. The right answer might depend on the workload more than on the architecture — workloads with hard precision needs may want Kostadis's checkpoints; workloads with high throughput may want Familiar's pipelines; workloads where the consumption pattern varies may want Kent's trained policies. A more useful future state might be a library of intelligence layers that compose, rather than a single winner.

**The OSS landscape is missing a layer that connects these systems.** Each of the three reinvents pieces of the same architecture above the same substrate. None of them exposes its intelligence layer as a library other systems could compose with. Familiar's reranking pipeline could be useful to Kent; Kent's APO training could be useful to Kostadis; Kostadis's CI invariant could be useful to jphein. But each system is bundled with its specific domain and substrate variant. The conversation in early 2026 about agent memory architecture has been mostly about which substrate wins; the conversation worth having for late 2026 is what the intelligence-layer interfaces look like and whether they can be portable across substrates.
